# C++ 进阶继承与多态学习笔记

本笔记对比并总结了 `@4_class_test` 中引入的进阶面向对象特性，包括多重继承的复杂性、静态多态的技巧以及解决菱形继承的方案。

---

## 1. 静态多态 vs 动态多态 (Static vs Dynamic Polymorphism)

代码展示了两种实现多态的路径：

| 特性 | 静态多态 (编译期) | 动态多态 (运行期) |
| :--- | :--- | :--- |
| **实现方式** | 函数重载、模板、CRTP | 虚函数 (Virtual Functions) |
| **分派机制** | 编译器根据类型静态绑定 | 通过虚表 (vtable) 动态查找 |
| **性能** | **零开销**，支持内联优化 | 有指针跳转开销，阻碍内联 |
| **灵活性** | 语法要求匹配（鸭子类型） | 必须有共同的基类接口 |

### 1.1 CRTP (奇异递归模板模式)
这是一种特殊的静态多态技巧。派生类继承自以自身为模板参数的基类。
```cpp
template<typename Derived>
struct Base {
    void interface() { static_cast<Derived*>(this)->implementation(); }
};
struct Son : Base<Son> {
    void implementation() { /* ... */ }
};
```
**优点**：实现了多态的语义，但避免了虚函数的性能开销。

## 2. 多重继承 (Multiple Inheritance)
C++ 允许一个类同时继承多个基类。
- **语法**：`class Son : public Base1, public Base2 { ... };`
- **风险**：如果 `Base1` 和 `Base2` 有同名成员，调用时会产生二义性，必须使用 `Base1::` 明确指定。

---

## 3. 菱形继承与虚继承 (Diamond Problem & Virtual Inheritance)

### 3.1 菱形继承的危机
当 `Father1` 和 `Father2` 都继承自 `GrandFather`，而 `Child` 同时继承这两个父亲时，`Child` 对象中会存在**两份** `GrandFather` 的成员。
- **问题**：资源浪费、逻辑二义性。

### 3.2 虚继承的解决方案
通过在继承时使用 `virtual` 关键字，确保在整个继承体系中只存在一份共享的基类对象。
```cpp
class Father1 : virtual public GrandFather { ... };
class Father2 : virtual public GrandFather { ... };
class Child : public Father1, public Father2 { ... };
```
**底层原理**：
- 虚继承的对象中包含一个**虚基类指针 (vbptr)**，指向**虚基类表 (vbtable)**。
- 无论继承链路多复杂，所有虚继承的子类都通过偏移量访问同一个 `GrandFather` 实例。

---

## 4. 模板编程 (Template Meta-programming)
代码中的 `call_run<T>(T &obj)` 演示了模板的本质：**代码生成器**。
- 编译器在编译期根据传入的 `RunnerA` 或 `RunnerB` 实例化出不同的函数版本。
- 这要求 `T` 必须具备 `run()` 方法，否则编译失败（这种约束在 C++20 中通过 `Concepts` 得到了更好的支持）。

---

## 5. 总结与最佳实践
- **优先组合而非继承**：多重继承容易使类关系变得混乱，只有在确实存在“既是 A 又是 B”的语义时才使用。
- **警惕菱形继承**：一旦出现菱形继承，务必使用虚继承，并仔细考虑初始化列表的顺序（虚基类由最底层的派生类负责初始化）。
- **性能敏感场景**：如果多态调用非常频繁且接口固定，考虑使用 **CRTP** 或 **模板** 替代虚函数。
